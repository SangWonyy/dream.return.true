INTRO

우리들의 쓰레기의 줄임말로, 쓰레기에 대한 관심을 친근감 있게 다가가기 위해 지어지게 되었다.
전세계 많은 인구에 의해 많은 쓰레기가 배출되고, 관리가 점점 시급해지고 있다.
이를 위해 WWOOSS 팀은 자신의 버린 쓰레기가 어떤 영향을 주고,
자신이 쓰레기를 어떻게 버려야하는지에 대해
손쉽고 재미 요소와 함께 깔끔한 UI/UX를 제공하여 쓰레기에 대한 관심을 끌기 위한 앱이다.

2. 어플리케이션 설계

- Multi-module 아키텍처
  • api : Controller, Service의 중심 비즈니스 로직 구현
  • batch : batch service를 위한 모듈. 예) 알람 일괄 발송
  • common : 공통 사항을 담은 모듈. 예) util, dto
  • domain-es : es 모듈. es document, repository 등 구성
  • domain-rds : rds 모듈. entity, repository 등 구성
- Why Multi Module
  • 증가하는 요구사항 및 기능
  • 모듈별 코드 배포가 가능
  • 모듈별 관리로 인해 멀티프로젝트 관리로 안해도 된다. 이에 따른 중복 코드 제거
  • 명확한 역할 분기로 인해, 코드 유지보수 관점에서의 장점을 가짐
- 도입 후 단점 생각
  • 초기 단일 프로젝트, 단일 모듈에서 옮기는 과정 속 많은 시간소비
  • 단방향 디펜던시을 유지하기 위한 노력
  • 각 모듈의 기능을 침범하지 않기 위한 노력
정리 : 적응 초반, 단일 모듈과 같이 단순히 기능을 추가 하는 것이 아닌, 해당하는 모듈에 기능을 추가해주어야했다. 이에 따른 디펜던시에 영향을 고민 또한 많이했다. 오히려 시간 소비가 많이 되었지만, 적응 후엔 전반적인 코드 구성을 바라보는 데에 있어 명확해지고 깔끔해진 결과를 가져왔다.

2. 인프라 설계

- 개발
  • 비용을 고려한 간단한 구조
  • 단일 인스턴스 내부 도커를 통해 api • swagger • db • es 생성 고려
  • 문제 - aws 프리티어 수준에서 실행하기 부족한 스팩
  • 해결 - 프리티어를 최대한 활용한 분리
  • (api • swagger) + (es) + (rds) 분리 후 비용은 es에서만 부과
- 운영 1차
  • 운영에서 가장 기본이라고 생각하는 구조를 설계
  • 2개의 가용영역
  • 각 가용영역에 1 public subnet • 2 private subnet
  • 외부 인터넷과의 통신을 위한 NAT
  • S3, CloudFront를 위한 endPoint
  • private subnet 접근을 위한 SSM
문제
- 기본이라고 생각을 가지고 설계 후 하나씩 도입하는 과정 속에 계속되는 추가비용.
- 사이드프로젝트를 고려하면 과할 수 있다고 판다.
- 추후 서비스 오픈 후 유저가 확보 될 시 점진적 확장으로 변경.

- 운영 2차
  • 개발과 동일하게 public subnet에 인스턴스를 할당
  • 개발 인스턴스 보다 상향된 인스턴스로 구성
